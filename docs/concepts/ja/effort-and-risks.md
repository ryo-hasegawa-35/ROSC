# 工数とリスク

## 前提

以下の概算は、次のような条件を前提にしています。

- Rust に強いシニアエンジニア 1 名がコアを主導する
- ダッシュボード向けに必要に応じてフロントエンド支援が入る
- UE5 統合では必要に応じて C++ 支援が入る
- CI、テスト、ベンチマーク整備を含める
- 単なる試作ではなく、現場投入を見据えた品質を目標にする

## 概算工数

以下は実装時間の目安であり、日程の保証ではありません。

| フェーズ | 内容 | 概算時間 |
| --- | --- | ---: |
| 00 | 仕様整理、テスト、ベンチ、リポジトリ土台 | 40-80 時間 |
| 01 | OSC コアプロキシとルーティング | 120-200 時間 |
| 02 | ダッシュボード、キャッシュ、リプレイ、可観測性 | 120-180 時間 |
| 03 | アダプター、ディスカバリ、メタデータ | 160-260 時間 |
| 04 | プラグイン、Wasm、スキーマ、コード生成 | 160-300 時間 |
| 05 | 共有メモリ IPC、UE5 / TD ネイティブ統合 | 220-400 時間 |
| 06 | セキュリティ拡張、同期、配布ハードニング | 180-320 時間 |
| 合計 | フルロードマップ | 1000-1740 時間 |

実際の感覚としては:

- 集中的に進めて 6-10 人月
- 現場要件を拾いながら進めるなら 9-15 か月程度

## 高リスク領域

### 1. 共有メモリによるネイティブ統合

リスク:

- 最も安定化が難しい
- OS 差分が大きい
- デバッグ難度が一気に上がる
- TouchDesigner / UE5 プラグインは初回実装後も保守負荷が続く

緩和策:

- IPC のバイナリプロトコルを最小限に保つ
- まずブローカー本体を先に出す
- ネイティブプラグインは後段の高速化オプションとして扱う

### 2. セキュリティ拡張

リスク:

- plug-and-play の採用性を損なう可能性がある
- 本番現場でロックアウトを起こす危険がある
- UX、フォールバック、復旧導線を丁寧に設計する必要がある

緩和策:

- optional mode にする
- secure / insecure の明確なプロファイルを用意する
- 「ブローカーで安全通信を終端し、下流は互換 OSC」の構成にする

### 3. スキーマとコード生成

リスク:

- 過剰設計になりやすい
- 早すぎる標準化で柔軟性を失いやすい
- UE5 / TD 利用者に明確な価値を出せないと重いだけになりうる

緩和策:

- 最初は lint とドキュメント用途から始める
- コード生成は、実際のユースケースが見えてから狭く導入する

### 4. Ableton Link と同期保証

リスク:

- この機能が入ると、ユーザーは非常に高い同期精度を期待する
- 実際にはネットワーク、OS スケジューリング、アプリ側挙動の制約が残る

緩和策:

- 実測値を可視化する
- 「タイムスタンプ伝播」と「完全同期保証」を分けて説明する

## 推奨体制

- ブローカーコア、parser、routing、tests、performance は 1 人が責任を持つ
- ダッシュボード、frontend、UX は part-time で担当を置く
- Phase 05 以降は native integration に part-time で担当を置く
- security と native integration を stable 扱いにする前に、少なくとも 2 種類の
  実運用に近いワークロードで検証する

## 追加で検討価値の高い機能

- route 単位の priority と backpressure policy
- config hot reload と validation / rollback
- OpenTelemetry または Prometheus export
- health check と heartbeat
- オフライン解析向けの traffic recording format
- synthetic traffic generator を含む simulation mode
- UE5、TouchDesigner、lighting、sensor 向けの preset profile
- crash-safe な config snapshot と automatic recovery
- なぜ packet が match / drop されたかを見せる rule debugger

## マイルストーン判定基準

次のフェーズへ進む前に、前フェーズで少なくとも以下を満たすべきです。

- 再現可能な benchmark がある
- 必要箇所に fuzz coverage がある
- Windows、macOS、Linux の CI が通っている
- 少なくとも 1 つの実運用寄りシナリオで有効性が確認できている
