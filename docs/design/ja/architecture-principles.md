# アーキテクチャ原則

## 1. 後方互換性を最優先にする

基本方針は「出力は厳格に、入力は寛容に」です。

- 出力側:
  - デフォルトでは OSC 1.0 準拠の UDP パケットを送出する
  - 元のアドレス文字列と引数順を保持する
  - 認証、発見、メタデータ、スキーマ情報を生 OSC ペイロードに強制的に
    埋め込まない
- 入力側:
  - 正しい OSC 1.0 メッセージとバンドルをそのまま受理する
  - type tag string を省略した古い実装にも頑健に対応する
  - 1.1 系の挙動は明示的に有効化されたときだけ使う

互換性ポリシー:

- 生の OSC 1.0 over UDP を不変の基準にする
- 1.1 的な追加機能は opt-in にするか、別トランスポートとして提供する
- 独自拡張は必ず交渉可能にするか、アダプター層に隔離する

## 2. OSC を「エンコーディング」として扱う

1.1 系資料の重要な点は、OSC を完全なプロトコルというより
コンテンツフォーマットとして捉え直していることです。

そのため:

- ルーティング、セキュリティ、発見、スキーマ、ダッシュボード、権限管理は
  上位レイヤーに置く
- コアブローカーは外部メッセージを内部の型付きイベントモデルへ正規化する
- アダプターは外部トランスポートの意味論と内部モデルの変換を担当する

こうすることで、既存 OSC に忠実でありながら、上位機能を安全に積めます。

## 3. 提案するプラグイン戦略

openFrameworks 的な「必要なものだけ追加する」発想は非常に良いです。
ただし 1 種類のプラグイン機構に寄せるのではなく、層で分けるのが良いです。

### Layer A: コンパイル時機能プリセット

Cargo feature と release profile で機能を束ねます。

- `core-osc`
- `dashboard`
- `discovery-mdns`
- `adapter-websocket`
- `adapter-mqtt`
- `filters-wasm`
- `integration-ue5`
- `integration-touchdesigner`
- `sync-ableton-link`

これにより、軽量バイナリとわかりやすい配布形態を両立できます。

### Layer B: 実行時 Wasm フィルタープラグイン

ユーザー定義のデータ変換は Wasm で実装します。

- smoothing
- scaling
- remapping
- sensor cleanup
- custom trigger logic

Wasm を使う理由:

- クロスプラットフォーム
- サンドボックス化しやすい
- ホットリロードと相性が良い
- Rust の不安定な ABI に依存しない

### Layer C: 外部アダプタープラグイン

より大きな拡張は、IPC や localhost 通信で接続する外部プロセス型
プラグインとして定義します。

向いている例:

- 独自ハードウェアブリッジ
- クラウド接続
- 専用コントロールパネル
- 現場固有ロジック

これは Rust のネイティブ動的ライブラリを直接ロードするより安全です。

### Layer D: 配布ビルダー / プロジェクトジェネレーター

openFrameworks の project generator に近い形として、目的別の配布物を
組み立てるビルダーを用意します。

- OSC-only build
- exhibition build
- browser-control build
- UE5 workstation build
- TouchDesigner kiosk build

これは拡張機構そのものではなく、配布・導入体験の層です。

## 4. ゼロトラストが OSC 文化と衝突しやすい理由

ゼロトラスト自体が悪いわけではありません。摩擦が生じるのは、現場の OSC の
使われ方がかなり軽量だからです。

摩擦が起きやすい理由:

- OSC は特に UDP 前提で fire-and-forget 的に使われることが多い
- 既存ツールの多くは認証ハンドシェイク、セッション状態、トークン更新を
  前提にしていない
- OSC 仕様自体に標準的なセキュリティ意味論、能力交渉、ID 交換がない
- ショー現場では「IP だけ差し替えてすぐ動かす」が優先されやすい
- 認証情報を生メッセージへ混ぜると互換性を壊しやすい

推奨方針:

- 生 OSC はそのまま使えるよう残す
- ゼロトラストはオプションのトランスポートまたは名前空間ゲートウェイとして
  追加する
- project ID、署名付きエンベロープ、ACL、レート制限はブローカーや
  アダプター層で処理する
- レガシー機器へは、ブローカーが安全な通信を終端して互換 OSC を再送する

これなら共有ネットワーク対策をしつつ、既存ワークフローも壊しません。

## 5. アドレスパターンの扱い

互換モードごとに分けて扱うべきです。

- `osc1_0_strict`
  - `?`, `*`, `[]`, `{foo,bar}`
- `osc1_0_legacy_tolerant`
  - type tag string 省略を受け入れる
  - address ベース routing は維持する
  - argument-aware transform は route の明示対応を要求する
- `osc1_1_extended`
  - `//` を追加

理由:

- 1.0 の挙動をデフォルトにする方が安全
- 1.1 論文でも `//` は曖昧だった挙動の明確化として導入されているため、
  ルート単位または名前空間単位の opt-in が望ましい

## 6. トランスポート方針

トランスポートは追加的に広げます。

- UDP データグラムを最重要経路として扱う
- TCP ストリームでは OSC 1.0 の size-prefix framing を互換モードとして維持
- ストリーム系トランスポート向けに、1.1 推奨に沿って SLIP framing も扱う
- Serial、USB、IPC はアダプター越しに提供し、コアのメッセージモデルを
  歪めない

## 7. クロスプラットフォーム方針

Windows、macOS、Linux は十分狙えます。

Rust で比較的まっすぐ実現できるもの:

- parser / encoder
- routing core
- UDP / TCP / WebSocket / MQTT
- Wasm runtime
- CLI
- ダッシュボードバックエンドの大部分

追加のプラットフォーム差分が大きいもの:

- mDNS / DNS-SD の挙動差
- 共有メモリ IPC の実装差
- インストーラ、署名、notarization
- システムサービス統合
- UI シェル統合
- Ableton Link の依存管理

結論:

- クロスプラットフォーム対応は十分可能
- ただし「配布と運用のクロスプラットフォーム化」は独立したフェーズとして
  きちんと扱う必要がある
